# bus_margin_stress_test — 버스·TE 구성 및 Golden Mismatch 이슈 메모

## 1. 목적

`replay.c`의 `bus_margin_stress_test`에서 chiplet 0 경로의 RBDMA tsync 태스크 설정(대략 7334~7342행 근처)과, 동작 시 관측되는 **데이터 golden mismatch** 현상을 문서화한다.  
(파일명 `missmatch`는 기존 표기를 유지한 것이며, 의미상으로는 **mismatch**와 동일하게 본다.)

## 2. 코드 기준: `generate_rbdma_task_tsync` 호출

`hal/rbdma/rebel/rbdma_if.c`의 시그니처:

```c
void generate_rbdma_task_tsync(uint32_t task_id, uint64_t src_addr, uint64_t dst_addr,
    uint32_t transfer_size, uint32_t split_granule, uint8_t task_type,
    uint32_t te_mask, uint32_t is_tsync);
```

`bus_margin_stress_test`(chiplet 0 분기)에서의 호출 예:

```c
generate_rbdma_task_tsync(0xDD3A0000, src_addr0, dst_addr0, trs_size, 0x7, 0, 0xf, 1);
generate_rbdma_task_tsync(0xDD3A0001, src_addr1, dst_addr1, trs_size, 0x7, 0, 0xf0, 0);
```

- `te_mask`는 디스크립터의 `run_conf1.te_mask`에 반영된다.
- `is_tsync == 1`인 첫 번째 태스크는 tsync 관련 필드(`tsync_conf_mask`, `sync_conf_gidx`, `intr_disable` 등)가 추가로 설정된다.

## 3. TE 마스크 의미 (요약)

| 호출 | `te_mask` (16진) | 의미 (비트 마스크) |
|------|------------------|---------------------|
| 첫 번째 | `0xF` | TE0, TE1, TE2, TE3 사용 |
| 두 번째 | `0xF0` | TE4, TE5, TE6, TE7 사용 |

- 시스템에 **TE는 총 8개**가 있다고 본다.
- 두 태스크는 **서로 다른 TE 그룹**(하위 4개 vs 상위 4개)을 사용한다.

## 4. 버스 채널과 TE 매핑 (전제)

버스는 **4개 채널**로 구성되며, 다음과 같이 대응한다고 정리한다:

| Bus channel | 담당 TE (전제) |
|-------------|----------------|
| ch0 | TE0, TE4 |
| ch1 | TE1, TE5 |
| ch2 | TE2, TE6 |
| ch3 | TE3, TE7 |

즉, 한 채널에 **서로 다른 tsync 태스크**가 겹쳐서 올라가 **동시에** 트래픽을 낼 수 있는 구조로 이해한다.

## 5. 동기·실행 순서 (chiplet 0 루프 안)

의도된 흐름(개념):

1. 두 개의 `generate_rbdma_task_tsync`로 **태스크 디스크립터를 큐에 적재**한다.  
   - 태스크 A: `te_mask = 0xF` (TE0~3), `is_tsync = 1`  
   - 태스크 B: `te_mask = 0xF0` (TE4~7), `is_tsync = 0`
2. `bus_margin_stress_iter_sync_before_kick()` 등으로 **다른 chiplet과의 단계 정렬**(있을 경우).
3. `dnc_autofetch_invoke(...)` — TU 측 동작.
4. `rbdma_tsync_by_cp()` — **CP 측에서 tsync에 따라** 준비된 태스크들이 **한 번에** 진행되며, 결과적으로 **8개 TE가 함께 동작**하는 구조로 본다.

정리: **두 함수 호출은 “큐잉” 단계**이고, **실제 동시 기동/동작은 `rbdma_tsync_by_cp()` 쪽에서 묶여** 일어난다고 이해하면 된다.

## 6. 관측 현상 (요약)

- `bus_margin_stress_test` 수행 시 **데이터 golden mismatch**(기대값과 DRAM/비교 결과 불일치)가 발생한다.
- **버스 클럭을 낮추면** 동일 시나리오에서 **재현되지 않는다**.

### 6.1 불일치 패턴·칩렛·빈도

| 항목 | 내용 |
|------|------|
| **에러 형태** | mismatch 시 **불일치는 1bit error**로 관측된다 (대량 랜덤 손상이 아님). |
| **발생 칩렛** | **chiplet 0, chiplet 1**에서 발생. (다른 칩렛은 본 이슈와 동일 패턴으로 우선 보고되지 않음.) |
| **상대 빈도** | **chiplet 0 > chiplet 1** (CL0에서 더 자주/빨리 관측). |
| **주소·채널 분포 (실험)** | mismatch 발생 시 **불일치 주소를 채널(§4 전제: TE–ch 매핑)으로 귀속**해 보면, **특정 채널에만 국한되지 않고** **버스의 모든 채널(ch0~ch3)**에 해당하는 영역에서 **관측된다**. (한 스트림/한 TE 그룹만의 문제로 보기 어렵다는 신호.) |

### 6.2 유사 테스트와의 관계

| 구분 | 설명 |
|------|------|
| **(A) `rbdma_shm_2_dram_tsync`** | **칩렛별로만** 동작시켜도 **§6.1과 동일한 증상**(1bit·CL0/CL1·빈도 경향)이 난다. |
| **(B) `bus_margin_stress_test`** | (A)에 **ncore unit 부하를 주기 위한 stress**를 가한 형태이다. |
| **(A) vs (B)** | **결과(증상)는 (A)와 동일**하다 — 즉 **stress 가미 자체가 실패 모드를 바꾸거나 새 유형을 추가한다고 보기 어렵다** (동일 메커니즘으로 수렴). |

### 6.3 위 관측이 시사하는 것 (요약)

- **1bit + 버스 클럭 민감**은 **넓은 버스트 손상**보다 **마진·SI/PI·샘플링 타이밍** 쪽 가설과 잘 맞는다.
- **CL0 > CL1**은 **물리 경로·전압·온도·SoC 내 위치**에 따른 **비대칭**을 의심할 근거가 된다.
- **`rbdma_shm_2_dram_tsync` 단독 재현**은 **ICI·멀티칩렛 동기·autofetch·CL2 perf 등 `bus_margin_stress_test` 전용 요소가 1차 원인일 가능성을 낮춘다.** 공통 분모는 **RBDMA tsync + SHM→DRAM 경로**이다.
- **`bus_margin_stress`의 ncore 부하는 실패 양상을 (A)와 구분하지 못함** → **지배적 실패 경로는 (A)와 동일한 경로**로 보는 것이 타당하다 (부하는 “노이즈/마진”만 더할 수 있으나, **동일 증상**이면 **근본 원인 후보는 (A)와 공유**).
- **전 채널 분포(§6.1)** → **“특정 채널만 과부하·단일 TE 그룹만 오동작”**만으로는 설명하기 어렵고, **버스/클럭/전원 등 공유 자원** 또는 **전 구간 공통 마진** 쪽 가설과 더 잘 맞는다.

## 7. 원인 가능성 검토 (가설) — 관측 반영 후 재정리

아래는 **§6 반영 후** 우선순위를 다시 잡은 가설이다. 실측·트레이스로 수정한다.

### 7.1 상대적으로 **가능성 높음** (1bit·클럭 민감·(A) 공통 경로와 정합)

1. **데이터 경로 SI/PI·타이밍 마진 (버스/DRAM I/F)**  
   1bit error, 클럭 인하 시 비재현은 **setup/hold**, **reflection**, **전원 노이즈** 등 **아날로그/마진** 계열과 정합. RBDMA→메모리 구간이 공통.

2. **RBDMA·메모리 컨트롤러 쪽 클럭/페이즈 의존 동작**  
   FIFO·크로스 도메인·리턴 패스에서 **단일 비트**가 깨질 수 있는 전형적 패턴. (A)와 (B) 모두 동일.

3. **chiplet 0 vs 1 비대칭 (물리·전원·온도·경로 길이)**  
   CL0에서 더 잦다면 **해당 칩렛의 DRAM/NoC 슬라이스·전원 도메인** 차이를 RTL/보드와 대조할 가치가 있다.

### 7.2 **유지하되 우선순위 조정**

4. **버스 포화 / 4채널·8TE 경합**  
   여전히 가능하나, **(A)만으로도 동일 증상**이면 “`bus_margin_stress`만의 과도한 동시성”보다는 **기본 tsync 부하에서도 터지는 마진 문제** 쪽으로 무게를 둔다.  
   **추가(전 채널 관측)**: 불일치가 **ch0~ch3 전부**에서 나타나면, **“한 채널만 병목”**보다는 **동시에 여러 채널이 트래픽을 낼 때 공통으로 깨지는** 패턴(공유 클럭·전원·글로벌 NoC 슬라이스 등)을 함께 의심하는 것이 자연스럽다.

5. **TE 마스크·채널 매핑과 RTL 불일치**  
   가능하나, **1bit·클럭 민감**이면 **설계 오류(완전 잘못된 연결)**보다는 **마진** 쪽을 먼저 본다.  
   **추가(전 채널 관측)**: **단일 채널에만 데이터가 잘못 연결**된 전형적 RTL 버그라면 **해당 채널에만** 집중된 오류가 기대되는 경우가 많다. **모든 채널에서 관측**되면, **“매핑 자체가 틀려서 한 채널만 깨진다”**는 단순 가설의 설득력은 **낮아진다**. 다만 **매핑은 맞고 각 채널 모두 타이밍 마진이 부족**한 경우에는 **전 채널에서 1bit**가 나올 수 있으므로, **마진·SI/PI** 가설과는 여전히 정합된다.

### 7.3 상대적으로 **가능성 낮아짐** (관측과의 정합이 약함)

6. **순수 SW 캐시 일관성 오류(비트 단위 1bit만)**  
   가능은 하나, **클럭 인하로 사라짐**과 **(A) 단독 재현**이 동시에 맞기 어렵다 → **우선 HW/펜스 검증** 전에 **캐시 무효화·완료 순서**를 재확인하는 것은 여전히 유효하나, **단독 원인**으로 보기엔 신호가 약해진다.

7. **ICI·멀티칩렛 동기·CL2 perf가 1차 원인**  
   **(A) 칩렛 단독**에서 동일하면 **1차 원인에서 제외**에 가깝다. (부차적 악화 요인은 남을 수 있음.)

8. **“대량 오탐(비트 패턴 무작위)”**  
   **1bit** 관측으로 **가설이 좁혀짐** — 여전히 **ECC/짧은 이벤트**와 구분은 필요.

### 7.4 정리

- **근본 후보**: **RBDMA tsync + SHM↔DRAM 데이터 경로**의 **전기적/타이밍 마진** (및 칩렛별 비대칭).  
- **`bus_margin_stress_test` 특유 요소**는 **증상을 바꾸지 못했으므로**, 디버깅 시 **(A) `rbdma_shm_2_dram_tsync`를 최소 재현축**으로 두는 것이 유리하다.

### 7.5 전 채널 분포 관측 반영 (재검토 요약)

| 관점 | 재검토 결과 |
|------|-------------|
| **글로벌 vs 단일 채널** | mismatch 주소가 **모든 채널(ch0~ch3)**에 걸쳐 보이면, 원인을 **“한 채널·한 TE 그룹만”**으로 좁히기 어렵다. **클럭/전원/공유 버퍼·전역 마진**류 가설(§7.1)의 상대적 비중이 **올라간다**. |
| **§8.10 TE 마스크 스윕** | 스윕은 **부하 패턴·동시성**을 바꿔 재현율을 보는 데는 여전히 유효하나, **“어느 채널 하나만 고장”**으로 귀결시키려는 기대는 **완화**하는 것이 맞다. 해석 시 **채널 단위 단정**보다 **전송이 겹칠 때의 마진/노이즈** 쪽을 병행한다. |
| **경합(§7.2-4)** | **4채널이 동시에 활발**할 때만 터지는지, **부분 마스크로 채널 수를 줄였을 때** 재현이 사라지는지와 **전 채널 분포**를 함께 보면, **순수 “한 사이드만 혼잡”**인지 **“풀 로드 시 전역 악화”**인지 구분에 도움이 된다. |
| **남는 불확실성** | 주소→채널 귀속은 **§4 매핑 전제**에 의존한다. RTL/주소 디코더와 다르면 해석을 수정해야 한다. |

### 8.10 제안 Trace: `te_mask` 조합 스윕 (`rbdma_shm_2_dram_tsync` 기반)

**의도**: 채널·TE별 기여도를 나누어 **어느 TE 그룹에서 1bit가 유발되는지** 좁힌다. (§4 TE–채널 매핑과 연계.)

| 검토 | 내용 |
|------|------|
| **타당성** | 높음. 현재 기본 조합은 `0xF` + `0xF0`(두 스트림). **부분 마스크**로 재현율이 달라지면 **경합·동시성·부하 패턴** 원인에 무게를 둘 수 있다. **다만** 실험상 mismatch가 **모든 채널 주소 범위에서 관측**되면, 스윕 목표를 **“단일 채널만 고장”으로 좁히기**보다 **전역 마진/공유 자원**과 **교차 검증**하는 용도로 보는 편이 낫다 (§7.5). |
| **구현 시 주의** | `replay.c`의 `rbdma_shm_2_dram_tsync`(tsync 경로)는 **DCL0 SHM→DRAM**·**DCL1 SHM→DRAM** **두 개의 독립 tsync 태스크**를 연속 큐잉한다. 조합 실험 시 **(a)** 두 태스크 모두에 **동일한 te_mask 정책**을 적용할지, **(b)** 한 스트림만 켜고 다른 하나는 **비활성/더미**할지 **실험 매트릭스**를 먼저 정한다. |
| **마스크 예시 (비트 = TE 인덱스)** | `0x0F`: TE0~3만 / `0xF0`: TE4~7만 / `0x11`: TE0+TE4만(ch0만) / `0x22`: TE1+TE5만(ch1만) … 문서 §4와 맞춰 표 작성. |
| **주의** | RTL·시퀀서가 허용하는 **유효한 te_mask** 조합이 제한될 수 있음 — **불가 조합**은 시뮬/담당자 확인. `is_tsync`(첫 태스크 1 / 둘째 0) 조합도 **한 스트림만 tsync** 실험 시 변경 여부 검토. |
| **추가** | 동일 **전송량·클럭**에서 **TE 수만 줄인 경우** vs **8TE 풀** 비교 → **동시성** 이슈인지 **절대 타이밍** 이슈인지 구분에 유리. |

### 8.11 제안 Trace: SHM src 골든 백업 후 mismatch 시 src/dst 귀속

**의도**: mismatch 시 **읽기 경로(SHM)** 문제인지 **쓰기 경로(DRAM)** 문제인지 분리한다.

| 검토 | 내용 |
|------|------|
| **타당성** | 높음. `basic_test_compare(src, dst)` 실패만으로는 **DMA가 읽은 src와 쓴 dst 중 어디서 비트가 틀렸는지** 직접 알 수 없음. **DMA 전** SHM 스냅샷과 비교하면 분석이 가능해진다. |
| **절차 (권장)** | ① `rand_fill` 직후 **src0/src1 전 구간**을 **CPU 접근 가능한 백업 버퍼**로 `memcpy` (또는 복사 루틴). ② **복사 후** `cpu_flush_dcache_range`(백업 + 필요 시 SHM). ③ `rbdma_tsync_by_cp()` 등 DMA 완료 후 **펜스**. ④ **SHM**: `basic_test_compare(src, golden_backup)` — 변하면 **src가 DMA 이후 훼손**되었거나(드묾), **읽기 관측** 문제. ⑤ **DRAM**: `basic_test_compare(dst, golden_backup)` — **기대 패턴과 dst** 직접 비교로 **쓰기 경로** 이슈에 집중. ⑥ 기존 `basic_test_compare(src, dst)`는 **보조**로 유지. |
| **해석** | src≈golden, dst≠golden(동일 오프셋) → **DRAM 쓰기/리턴** 쪽 의심. dst가 golden과 일치하는데 src≠dst면 **비교 순서/캐시** 재확인. **둘 다** 틀리면 **패턴 재현** 필요. |
| **주의** | 백업 버퍼가 **DRAM/SHM과 겹치지 않게**. DMA 완료 전 **백업을 덮어쓰지 않음**. **DCL0/DCL1 두 스트림**이면 **백업도 두 영역** 각각. |

### 8.12 위 두 항목 외 보완 (추가 권장)

- [ ] **실패 시점 로그**: `basic_test_compare(src0,dst0)` vs `src1,dst1` 중 **어느 쌍에서 먼저 실패**하는지 매 실행 기록 (이미 §8.3과 연계).
- [ ] **백업 실험 + TE 스윕**: 가능하면 **동일 TE 조합**에서만 src/dst 귀속 실험을 반복해 **원인 축**을 교차 검증.
- [ ] **소프트 카피 백업 한계**: 1bit가 **DMA 엔진 내부 파이프**에서만 발생하면 src/dst 메모리는 둘 다 “정상”으로 보일 수 있음 — 그 경우 **인라인 로직·하드웨어 trace**가 필요하다는 **한계**를 문서화.
